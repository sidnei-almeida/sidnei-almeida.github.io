3) DATA FLOW FOR FRONTEND
-------------------------

Recommended flow (standard paper endpoints):
1. Seed balance for user
2. Create signal or process inline signal
3. Refresh summary + positions + trades + equity-history + signals
4. Render KPI cards, table data, and equity chart

Recommended flow (mocked wallet endpoints):
1. Connect test mode
2. Seed/reset/process via /api/test-mode/paper/*
3. Pull all dashboard data using /api/test-mode/paper-dashboard


4) ALL ENDPOINTS + HOW TO IMPLEMENT IN FRONTEND
------------------------------------------------

4.1 PHASE 2 MAIN ENDPOINTS (prefix: /api/paper)
-----------------------------------------------

[Portfolio]

GET /api/paper/portfolio/{user_id}
- Purpose: full simulated portfolio row
- Use when: you need base portfolio state
- Frontend: fetch once on page load or after mutations

GET /api/paper/portfolio/{user_id}/summary
- Purpose: KPI cards payload
- Returns: initial_balance, current_cash, total_equity, total_return_pct, total_trades, win_rate, etc.
- Frontend: map directly to summary cards

GET /api/paper/portfolio/{user_id}/positions
- Purpose: open positions table
- Frontend: render symbol, quantity, avg_entry_price, current_price, unrealized_pnl

GET /api/paper/portfolio/{user_id}/trades?limit=200
- Purpose: trade history table
- Frontend: render side, qty, entry, exit, realized pnl, status, opened_at/closed_at

GET /api/paper/portfolio/{user_id}/equity-history?limit=500
- Purpose: line chart source
- Returns: recorded_at, equity, cash_balance
- Frontend: x=recorded_at, y=equity

[Signals]

POST /api/paper/signals
- Purpose: create signal first, process later
- Body:
  {
    "symbol": "AAPL",
    "signal_type": "BUY",
    "signal_price": 180.5,
    "confidence_score": 0.8,
    "explanation": "optional text"
  }

GET /api/paper/signals?limit=100
- Purpose: global signal list (plus user-scoped where applicable)
- Frontend: signal history table

POST /api/paper/signals/process/{user_id}
- Purpose: process existing signal by ID
- Body:
  {
    "signal_id": 1
  }
- Frontend: after success, refresh summary/positions/trades/equity/signal list

[Simulation]

POST /api/paper/simulation/{user_id}/seed-balance
- Purpose: create/reset paper portfolio with initial balance
- Body:
  {
    "initial_balance": 10000
  }

POST /api/paper/simulation/{user_id}/reset
- Purpose: full simulation reset (positions/trades/equity history)
- Body:
  {
    "initial_balance": 10000
  }

POST /api/paper/simulation/{user_id}/process-signal
- Purpose: create and process signal in one call
- Body:
  {
    "symbol": "AAPL",
    "signal_type": "SELL",
    "signal_price": 185.0,
    "confidence_score": 0.72,
    "explanation": "optional"
  }

[Extra]

POST /api/paper/signals/mock-batch
- Purpose: insert sample signals for quick testing

5) FRONTEND IMPLEMENTATION CHECKLIST
------------------------------------

A) Setup
- Keep API base URL in config (e.g., VITE_API_URL)
- Add auth token storage and Authorization header helper for test-mode endpoints
- Add a mode state in frontend:
  - wallet_mode = "connected" | "mocked"

B) Dashboard rendering
- KPI cards from summary endpoint
- Open positions table from positions endpoint
- Trade history table from trades endpoint
- Signal history table from signals endpoint
- Equity line chart from equity-history (or mocked dashboard combined payload)

C) Actions
- Seed button -> seed-balance endpoint
- Reset button -> reset endpoint
- Process Signal button/form -> inline signal or signal-by-id endpoint
- After each mutation: refetch all dashboard data

D) Error handling
- Handle 401: redirect to login / request token refresh
- Handle 404 for non-seeded portfolio: prompt seed action
- Handle 400 validation errors: show backend message directly in UI

E) Polling strategy (optional)
- If real-time-like UX is needed:
  - refresh summary + positions + trades every 3-5 seconds
  - refresh equity-history less often (e.g., every 10-20 seconds)
